## Findings

### ISSUE 1 — The URL-compatibility gate measures path parity within the new host and silently ignores the hostname change [P1][P2]

- Confidence: 80
- Location: Section: 10. URL 兼容性 (interacting with Section 3 Goal #5, Section 7.2 Build job steps 6–7, Section 14 完成标准)

**What the spec promises.** Goal #5 (Section 3) commits to "现有公开网址、文章、标签、RSS、站点地图和静态资源继续可访问." Section 10 elevates URL continuity to a *blocking* release condition ("必须把 URL 兼容性作为发布阻断条件"). Section 14 lists "所有旧文章 URL 均继续可访问或正确跳转" as a completion criterion. So the spec explicitly *owns* URL continuity — this is not an external requirement I am importing.

**What the spec actually verifies.** Section 10's verification compares the *old* `sitemap.xml` URL list against the *new* `sitemap.xml` URL list by **path**, and satisfies each old URL with either a same-path page or a Hugo `aliases` redirect. Section 7.2 steps 6–7 only assert that the *new* output contains `ken-zy.github.io` and does not contain `jiadingyi.github.io`. Neither check touches the question that actually matters for "existing public URLs": **do inbound links to `https://jiadingyi.github.io/Tips/...` still resolve after this migration?**

**Why this is directional, not a detail.**
- `aliases` only generate redirect pages *within* `ken-zy.github.io/Tips/`. They cannot repair a hostname change — an external link to `jiadingyi.github.io/Tips/post/…` never reaches Hugo's alias machinery.
- The spec's path-level sitemap comparison normalizes the host away, so the gate can pass fully green while every historical external/inbound link and SEO result pointing at the old host is broken. That is a false-success gate: the checklist reports "URL 兼容通过" for a property it never measured.
- The whole design rests on an **unstated, load-bearing assumption**: that `jiadingyi.github.io` still redirects to `ken-zy.github.io`. That is true only if `jiadingyi → ken-zy` is a GitHub *username rename* (and even then only until the `jiadingyi` username is reclaimed by anyone). If `ken-zy` is a separate/new account, old-host links are permanently dead and nothing in the design detects it.

**Recommendation (concrete, direction-level).**
1. State the assumption explicitly. If this is a username rename, document `jiadingyi.github.io → ken-zy.github.io` as the load-bearing dependency and note the reclaim-risk expiry. If it is a separate account, add an explicit decision: either accept the loss of old-host inbound links (and move that out of Goal #5), or keep the old repo/account serving an HTTP redirect.
2. Make the host normalization in Section 10 explicit, so the gate is not read as guaranteeing host-level continuity when it only guarantees path parity on the new host.
3. Add one verification step that actually exercises the old host (fetch a representative `jiadingyi.github.io/Tips/...` URL and confirm it 3xx-redirects to the new host) — otherwise the "blocking condition" framing overstates what is checked.

## Pre-mortem (mandatory)

**Single most likely directional failure:** the migration completes fully green against its own checklist — site live at `ken-zy.github.io/Tips/`, old domain purged from output, path-level URL parity verified, Actions Build/Deploy both succeed — yet the stated goal "existing public URLs remain reachable" (Goal #5) fails for every external link pointing at the old `jiadingyi.github.io` host.

**Why the spec invites it:** the spec reduces the old host to a string to *delete* from build output (Section 7.2 step 6, Section 14) and scopes its only URL-compatibility analysis to Linux vs macOS **path case** (Section 10). Nowhere does it ask whether the host transition itself preserves access. The verification is therefore aimed at the wrong axis (path parity within the new host) than the one the goal cares about (old-host → new-host continuity). A green checklist will read as success while the real user-facing asset — inbound links / SEO to the historical URL — silently breaks.

## LGTM (on everything else)

The primary direction is sound and correctly proportioned to the real problem:

- **Problem framing [P1]:** the dedupe / manual-sync / blurred source-vs-generated boundary problem is real and concrete; consolidating source + publishing into the already-public `Tips` repo (preserving repo name and Pages URL) is the right target, and correctly avoids the trap of making `Tips_hugo_source` public (which would have changed the Pages URL).
- **Direction [P2]:** single repo + Hugo source in `site/` + GitHub Actions Pages-artifact deploy (no `gh-pages` branch) is the canonical, officially documented pattern; the design cites the correct Hugo/GitHub Pages docs and pins the supply chain (Action SHAs, Hugo `0.105.0` + SHA-256, PaperMod commit).
- **Alternatives [P3]:** the two realistic alternatives (keep two repos + automate sync; or promote `Tips_hugo_source` to public) are both inferior for stated reasons, and the design implicitly rejects them correctly.
- **Rollback:** the "keep root HTML until Actions deploy is verified, then a separate cleanup PR" structure gives a near-free rollback path and is the strongest part of the design.

## Below threshold (not blocking; confidence < 70, reported for transparency)

- The **dual workflow trigger states** (startup vs stable, Section 7.1) are the one place carrying self-imposed complexity — the cleanup PR must remember to rewrite trigger conditions, a manual error-prone step, purely to avoid noisy failed deploy runs while Pages source is still branch-based. It is defensible and cheap enough that I am *not* raising it as a YAGNI issue, but it is the first candidate to simplify if it causes friction (e.g. just gate the deploy job on the event rather than maintaining two trigger blocks). Confidence this is a real directional problem: ~40 — below threshold.
