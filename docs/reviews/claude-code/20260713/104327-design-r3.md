## Finding

### ISSUE 1 — The theme-swap acceptance gate measures "renders OK", not "matches the current live site" [P2]

- Confidence: 72
- Location: Section: 11 验证矩阵 (页面视觉 row, l.301) interacting with Section 4 非目标 (l.48), Section 5 D5 (l.100–102), Section 13 风险 (l.336)

**The claim.** D5 asserts the swap "不视为主题升级" (not a theme upgrade → implying no visual change), and Section 4 makes "调整网站视觉" an explicit **non-goal** — the migration must not alter appearance.

**Why the gate cannot back that claim.**
- `efe4cb45` was chosen by *date proximity* ("2023-10-08 更新主题前最接近的官方可达提交"), not by established rendering equivalence to the lost commit. And `c07d9ce` is **absent from official PaperMod history** (GitHub API: no commit found), which means the live theme's provenance is unverified — it may be a fork/customized or post-update commit. So `efe4cb45` could render the site differently (e.g., reverting a post-2023-10-08 theme change the live site currently reflects, or dropping a customization).
- The evidence D5/risk cite for the swap — "73 legacy paths preserved" — is a *content/URL* property driven by `content/` + `config.yml`; it is **independent of the theme** and says nothing about rendering fidelity.
- The only theme-relevant gate, the 页面视觉 row, passes on "…正常" (renders **OK**), not on "matches the current live rendering." A pre-update or divergent PaperMod can render perfectly "正常" while the site visibly changed — silently breaching the Section 4 non-goal, with every gate green.

**Why this is directional, not plan-level.** The issue is *what the acceptance criterion measures*, and the fix is available at the design level: the authoritative reference — the current live rendered output — already sits committed at the repo root (`index.html`, `assets/`, `post/`, …). The design already uses it as the URL baseline in Section 10; it should likewise be the fidelity reference for the theme swap. Reframe the gate from "pages render 正常" to "representative pages match the current root rendering (host-normalized)". That converts "不视为主题升级" from an assertion into evidence.

**Honest caveats (why 72, not higher).** PaperMod ships a small template set (home / list / single / taxonomy / terms), so a representative visual check plausibly exercises nearly every template; a full mechanical byte-diff is impractical as a gate because the old root output carries the old host and content-hashed asset filenames that would dominate the diff. The direction is therefore sound — the gap is the *reference* the visual check compares against, not the approach.

## Pre-mortem (mandatory)

Assume the spec is implemented exactly as written and it later "fails" directionally. The single most likely failure: **the migrated site goes live rendering correctly but not identically to the pre-migration site** — because `efe4cb45` is not the lost `c07d9ce` and the live theme's true provenance was never established — **and nobody notices**, because every gate only asked "does it render 正常?" rather than "does it match what was there before?" The spec invites this by (a) asserting "不视为主题升级" on evidence (73 preserved paths) that is independent of theme rendering, and (b) setting the visual gate's bar at "正常" while the authoritative reference rendering sits unused in the repo root. Result: a silent breach of the "不调整网站视觉" non-goal that passes the whole verification matrix.

## Verdict

The PaperMod replacement is **directionally sound and consistently scoped**; the remaining directional gap (ISSUE 1, confidence 72) is narrow — anchor the theme-swap fidelity gate to the current root rendering rather than an abstract "renders 正常" bar, so the "no visual change" non-goal is actually measured rather than asserted.
