## Finding

### ISSUE 1 — The tightened gate demands exact visual identity to a baseline built by an unrecoverable theme, with no adjudication path for the realistic "cosmetic difference" case [P2]

- Confidence: 72
- Location: Section: 5 D5 (l.102), Section: 9 Phase B step 6 (l.244), Section: 11 页面视觉 (l.303), Section: 14 完成标准 (l.358)

**What the fix now requires.** Every touchpoint sets a hard-zero bar: the new build (rendered by `efe4cb45`) must be visually identical to the committed root output (rendered by the lost `c07d9ce`), with *exactly two* permitted exceptions — hostname/link targets and build-year. Anything else is a visible difference that blocks the migration.

**Why this is a new directional problem.** The baseline is the output of a theme commit that is **irrecoverable and absent from official PaperMod history** (the R3 premise: "no commit found"), which makes a fork/customization plausible. `efe4cb45` is a *different* commit chosen by date proximity, not by demonstrated rendering equivalence. So a genuine-but-cosmetic third category of difference (e.g. a few-pixel spacing change, a tweaked CSS rule present only in the customized original) is a realistic outcome — yet the spec provides **no branch for it**. The design has only two states: identical (pass) or different (block). If a real cosmetic difference appears, the migration can neither complete (its own completion criterion fails) nor proceed except by an **undocumented ad-hoc waiver** — which silently reopens exactly the R3 ambiguity (differences accepted with no principled threshold). This is an over-correction: R1–R3 risked false-pass (green while wrong); this fix introduces a false-fail / deadlock risk (cannot go green even though the substitution is the correct, user-approved, unavoidable one).

**Why it is directional, not a minor edge case.** It is a structural property of the acceptance criterion (binary exact-match, no tolerance, no resolution fork) against a baseline whose exact theme can never be reproduced — it can block the entire migration or force an unprincipled exception at the completion gate. Same altitude as the R3 finding the author just accepted.

**Recommendation (directional).** Keep R3's rigor (differences must be *seen and judged*, never ignored) but add the missing branch: define what happens when a visible difference beyond host/build-year appears — either a bounded, explicit tolerance ("cosmetic differences that do not alter layout, readability, or interaction are recorded and accepted by the owner") or a decision fork ("any such difference triggers a documented owner sign-off, or a theme-source-recovery attempt, before release"). This converts the gate from "identical-or-deadlock" into "identical, or a difference the owner has consciously adjudicated."

**Honest caveats (why 72, not higher).** If `efe4cb45` in fact renders identically to the live site (plausible if the original was pristine PaperMod and the two commits are CSS-stable neighbours), the gate passes trivially and this never bites — and the spec's own "不视为主题升级" assertion expects exactly that. The structural gap is definitely present in the text; whether it triggers depends on an unverified equivalence. One could also argue block-and-decide-manually is implicitly acceptable — but the spec does not say so, and leaving it implicit is the gap.

## Pre-mortem (mandatory)

Assume the spec is implemented exactly as written and it "fails" directionally. Most likely failure: the representative-template comparison surfaces a small, real rendering difference between `efe4cb45` and the lost `c07d9ce` baseline (a spacing/typography nudge, or a customization missing from the official commit). Because the spec permits *only* host and build-year diffs and defines no adjudication path, the migration stalls at its own blocking gate over a difference that is cosmetically trivial and, given the original theme is gone forever, **unfixable to identity**. The realistic escape — the owner glancing at it and accepting it — is undocumented, so it happens as an off-spec waiver that quietly restores the R3 "undefined fidelity threshold" the whole chain of fixes was meant to eliminate. The spec invites this by hard-coding a two-item exception list and treating exact identity to an unrecoverable baseline as a completion criterion, without a branch for the most probable real outcome of its own forced theme substitution.

## Verdict

The R3 fix is correctly applied and consistently scoped; the core direction remains sound. One new directional issue (ISSUE 1, confidence 72): the tightened gate has no adjudication path for a cosmetic-difference outcome against a baseline whose theme is irrecoverable — add a bounded tolerance or an explicit owner-sign-off fork so a trivial, unfixable-to-identity difference cannot deadlock the migration or force an undocumented waiver.
