## 一、先用一分钟认识 Git 和 GitHub

**Git 是电脑里的版本记录器。**

它会记录项目每次发生了什么变化。可以把它想成游戏存档：Codex 修改文件前后，你都可以留下一个版本；以后改坏了，还能找到之前的内容。

**GitHub 是存放和分享这些版本的网站。**

Git 主要在你的电脑里工作；GitHub 保存一份远端版本，让你可以公开分享、换电脑继续使用，或者和别人合作。

两者的关系可以记成一句话：

> Codex 帮你干活，Git 帮你记账，GitHub 帮你保存和分享。

本篇使用真实项目 [ken-zy/BL75](https://github.com/ken-zy/BL75) 演示。BL75 是一个 Markdown 知识资料库；你也可以换成自己的文档、网页或代码项目。

## 二、在 Codex 中打开 BL75

在 Codex 左侧选择 `BL75`。如果列表里还没有它，就点“新建任务”，再从输入框上方选择 BL75 文件夹。

**告诉 Codex：**

> 先只读查看这个项目。告诉我里面主要有哪些文件，以及它目前有没有使用 Git 管理。不要修改任何内容。

**你要检查：** Codex 能说出主要文件，输入框上方显示 `BL75 · 本地`。

## 三、公开之前，先让 AI 检查隐私

公开 GitHub 仓库意味着任何人都能看到里面的文件。上传前最重要的不是点哪个按钮，而是确认哪些内容真的能公开。

**告诉 Codex：**

> 先不要修改或上传。检查这个项目里有没有私人笔记、密码、密钥、账号信息、本地临时文件和不应该公开的图片。不要读取任何密钥的具体内容。最后告诉我：哪些可以公开、哪些应该排除、哪些需要我确认。

**你要检查：** 每个准备上传的文件都在“可以公开”清单里；不确定的内容已经移走或明确排除。

**停下来确认：** 只要还有一个文件拿不准，就不要继续。让 AI 先排除它，不要抱着“应该没人看到”的想法上传。

## 四、让 Codex 准备项目说明和保护规则

正式记录版本前，先准备三个东西：

- `.gitignore`：告诉 Git 哪些临时文件永远不要记录
- `README`：告诉别人这个项目是什么、应该怎么阅读
- `LICENSE`：告诉别人可以怎样转载和修改内容

本案例排除了 macOS 自动生成的 `.DS_Store`，也排除了 Obsidian 自动创建的 `.obsidian/` 本地配置目录。这个目录可能记录界面布局和最近打开的文件，不属于知识库正文，不应该上传到公开仓库。

内容许可证选择 CC BY-NC-SA 4.0：别人可以署名进行非商业分享和改编，但改编后的内容也要使用相同许可证。

**告诉 Codex：**

> 帮这个项目准备公开仓库需要的说明：忽略系统临时文件和 Obsidian 本地配置目录，写一份简单 README 介绍项目用途和目录，并使用 CC BY-NC-SA 4.0 许可证。完成后先给我看修改摘要，不要保存版本，也不要上传。

**你要检查：** 项目根目录出现忽略规则、README 和许可证；`.DS_Store` 与 `.obsidian/` 不在准备上传的清单里；README 的项目说明准确；私人内容没有重新出现。

## 五、开启 Git 版本管理

接下来让 Git 开始管理这个文件夹。这个动作只发生在本机，不会自动把内容传到网上。

**告诉 Codex：**

> 为当前 BL75 文件夹开启 Git 版本管理，主分支使用 main。完成后告诉我 Git 准备记录哪些文件、忽略了哪些文件。先不要保存第一个版本，也不要上传。

这里的 **main** 可以理解成项目的“主线”。以后你看到 Codex 输入框上方显示 `main`，就知道当前正在这条主线上工作。

**你要检查：** Codex 明确告诉你 Git 已经开启；准备记录的清单中没有私人内容和系统临时文件。

## 六、检查修改，再保存第一个版本

Git 里的“提交”（commit），就是保存一个带说明的项目版本。它仍然只在本机，不等于上传。

不要让 AI 看都不看就提交。正确顺序是：先列出准备保存的文件，再总结具体改动，最后等你确认。

**先告诉 Codex：**

> 列出这次准备保存的全部文件，并用普通话总结每个文件发生了什么变化。检查有没有私人内容或不相关文件。先停在这里，等我确认，不要提交。

确认清单没有问题后，再说：

> 我确认这些文件可以保存。请创建第一个版本，说明写“初始化 BL75 公开资料库”。完成后告诉我版本是否保存成功，以及项目里还有没有未保存的修改。不要上传。

**你要检查：** Codex 告诉你第一个版本已经保存，并且没有遗漏的修改。

**停下来确认：** “帮我看看修改”不等于“允许提交”。只有你明确说“提交”或“保存这个版本”时，AI 才应该执行。

## 七、创建公开 GitHub 仓库

现在本地已经有一个完整版本，才轮到 GitHub。

把本地版本发送到 GitHub，叫作“推送”（push）。这一步会把文件传到互联网，所以必须明确告诉 AI：使用哪个账号、仓库叫什么、是公开还是私密。

**告诉 Codex：**

> 我确认当前已经保存的内容都可以公开。使用我已经登录的 GitHub 账号创建一个名为 BL75 的公开仓库，简介写“BL75 学习与实践资料库”，然后把当前主线推送上去。创建前先确认没有同名仓库；完成后验证网页能公开访问。本次禁止强制推送。

**你要检查：** 浏览器不登录也能打开 [https://github.com/ken-zy/BL75](https://github.com/ken-zy/BL75)，页面里能看到 README 和项目文件。

**停下来确认：** 一旦内容已经推送到公开仓库，即使后来删除文件，旧版本仍可能保留记录。公开之前的检查不能省。

## 八、以后怎样让 AI 管理 Git

以后每次修改项目，都按这个顺序和 Codex 合作：

1. **开始前**：让它告诉你当前项目有没有尚未保存的修改。
2. **做任务**：只让它修改这次任务涉及的文件。
3. **保存前**：让它列出改了哪些文件，并用普通话总结差异。
4. **你来确认**：删除不相关内容，确认没有隐私和密钥。
5. **明确提交**：你说“提交”后，它才保存一个新版本。
6. **明确推送**：你说“推送”后，它才把版本发送到 GitHub。

可以长期复用这段话：

> 开始前先告诉我项目当前状态。完成任务后，列出修改文件并用普通话总结变化。在我明确说“提交”之前不要保存版本，在我明确说“推送”之前不要上传。不要读取或提交密码、密钥、账号信息、私人笔记和系统临时文件，也不要强制推送。

## 九、Git 出问题时怎么办

- **AI 改坏了文件**：先不要继续修改，让它对比当前内容和上一个版本，再决定恢复哪些部分。
- **提交里混进了无关文件**：如果还没上传，让 AI 停下来重新整理；如果已经公开，先停止后续操作，再处理历史记录。
- **推送失败**：不要让 AI 强行覆盖远端。先让它解释本地和 GitHub 哪边发生了新变化，再决定怎么合并。
- **看不懂 AI 的说明**：直接要求“不要讲命令，用普通话解释会影响哪些文件”。

## 十、这一篇做完的标志

- [ ] 能用自己的话解释 Git 和 GitHub 的区别
- [ ] 知道提交是保存本地版本，推送才会上传到 GitHub
- [ ] 每次提交前都会先检查文件清单和修改摘要
- [ ] 能在 GitHub 打开自己的项目
- [ ] 知道 AI 负责执行，人负责检查和授权
